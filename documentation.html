Ticketing System

Technical & Operational Documentation — v1.0

A lightweight, self-hosted customer support ticketing platform built in PHP with a Bootstrap 5 interface. No framework, no Composer, no database required — runs on any standard PHP shared hosting out of the box.

Request Flow

  1. Browser loads index.php — PHP validates session and renders the HTML shell
  2. JavaScript in app.js calls inc/api.php?action=<name> via fetch()
  3. api.php validates session, checks role permission, executes the function
  4. Function reads/writes JSON files through helpers.php utilities
  5. JSON response returned; JS renders the result into the DOM
  6. On mutations (new ticket, comment, status change), mailer.php sends notifications
Path resolution: APP_ROOT is resolved with realpath() to handle symlinked shared hosting roots. APP_URL is derived from SCRIPT_NAME / SCRIPT_FILENAME rather than DOCUMENT_ROOT, making it symlink-safe on any server configuration.

Installation

  1. Upload all files to your server (FTP, SFTP, or Git clone)
  2. Make the data/ directory writable: chmod 755 data/
  3. Make the uploads/ directory writable: chmod 755 uploads/ (create it if missing)
  4. Visit https://yourdomain.com/path/to/index.php — login screen appears
  5. Log in with the default admin credentials (see Default Credentials)
  6. Go to ⚙️ Impostazioni to configure brand name, color, and email notifications
Security: Change all default passwords immediately after the first login. The data/ directory must not be publicly accessible via HTTP — add an .htaccess rule or configure Nginx to deny direct access.
No rewrite rules needed. The application works without mod_rewrite. All pages are accessed directly via their PHP filenames.

Roles & Access Control

The system has three roles with progressively restricted permissions:

Admin

Administrator

Full system access. Typically 1–2 people.

  • Manage all tickets (view, update status, priority, assignment, delete)
  • Manage all users (create, edit, activate/deactivate, delete)
  • View admin dashboard tabs: Tickets, Users, Report, Log Attività, Impostazioni
  • Access statistics report and full activity log
  • Export CSV: all tickets, activity log, statistics report
  • Configure white-label branding and SMTP email settings
  • Receive email notifications for all new tickets, status changes, and comments
Operator

Operator (Support Agent)

Handles ticket resolution. Multiple operators possible.

  • View all tickets in the system
  • Update ticket status and priority
  • Add comments to any ticket
  • View in-app notification badge for tickets updated in the last 24h
  • Receive email notifications for new tickets and user comments
  • Cannot: create/delete users, access admin report, delete tickets, change settings
User

Standard User (Customer)

End-user who submits support requests.

  • View only their own tickets
  • Create new tickets (title, description, category, priority)
  • Add comments on their own open tickets
  • Upload images as attachments on their tickets
  • Receive email notifications when an operator changes ticket status or adds a comment
  • Cannot: see other users' tickets, change status/priority, access admin area

🔴 Administrator Workflow

Daily Operations

  1. Log in → land on the Ticket tab showing all tickets in the system
  2. Review ticket list — filter by status (nuovo, in lavorazione, in attesa, risolto, chiuso) or priority
  3. Click any ticket to open the detail view — review description, history, and comments
  4. Assign unassigned tickets to an operator using the Assegna a dropdown
  5. Update status or priority as needed

User Management

  1. Click the 👤 Utenti tab
  2. To add a user: click + Nuovo Utente, fill in username, name, email, role, password
  3. To edit: click the edit icon on any user row — modify name, email, role, active status, or password
  4. To deactivate: toggle the Attivo field to false — the user can no longer log in
  5. To delete: click the delete icon (red trash) — confirms before removing

Monitoring & Reporting

  1. Click 📊 Report for an overview: total tickets, counts by status / priority / category
  2. Click 📋 Log Attività for a full chronological audit trail (paginated, 50 entries/page)
  3. Use the ⬇ CSV buttons on Report and Log tabs to export data for Excel

System Configuration

  1. Click ⚙️ Impostazioni
  2. Set Nome brand, Colore primario, and optionally a Logo URL — these apply site-wide immediately after save
  3. To enable email notifications: fill in SMTP credentials, toggle Abilita notifiche email, save
  4. After saving, the page reloads to apply branding changes

🟠 Operator Workflow

Handling Incoming Tickets

  1. Log in → ticket list shows all open tickets across all users
  2. Optionally check the 🔔 notification badge (top-right) for tickets updated in the last 24h
  3. Click a ticket to open the detail view
  4. Read the description and any existing comments
  5. Set status to in lavorazione to signal the ticket is being worked on
  6. Add a comment explaining next steps or requesting more information

Resolving a Ticket

  1. Investigate the issue and provide a resolution in the comment section
  2. Change status to risolto — the ticket opener receives an email notification
  3. If the user confirms, status can be moved to chiuso (by admin or operator)
  4. If the issue is not resolved, the user can reopen by adding a comment — status moves back

Status Reference

StatusMeaningWho sets it
nuovoJust created, not yet assignedSystem (automatic)
in lavorazioneOperator is actively working on itOperator / Admin
in attesaWaiting for info from the userOperator / Admin
risoltoIssue resolved, awaiting user confirmationOperator / Admin
chiusoFully closed — users cannot add commentsOperator / Admin

🔵 Standard User Workflow

Submitting a Support Request

  1. Log in → see only your own tickets in the list
  2. Click + Nuovo Ticket
  3. Fill in:
    • Titolo — short summary of the issue
    • Descrizione — full description with as much detail as possible
    • Categoria — type of issue (e.g., Supporto Tecnico, Fatturazione…)
    • Priorità — Bassa / Media / Alta / Urgente
  4. Submit — the ticket appears in your list with status nuovo
  5. All admins and operators receive an email notification of the new ticket

Following Up on a Ticket

  1. Click the ticket in your list to open the detail view
  2. Read operator comments in the timeline
  3. Add a comment to provide more information or ask a question
  4. Attach images if helpful (drag & drop or file picker)
  5. You will receive an email when the operator changes the status or adds a comment
Closed tickets: Once a ticket reaches chiuso status, comments are disabled. If the issue resurfaces, open a new ticket referencing the original ticket ID.

Ticket Lifecycle

Every ticket carries a history array — an immutable audit trail of all actions taken:

[
  { "action": "created",        "by": "u_xxx", "by_name": "Mario Rossi",  "at": "2026-04-17T..." },
  { "action": "status_changed", "by": "u_yyy", "by_name": "Op. Verdi",    "at": "...", "note": "nuovo → in lavorazione" },
  { "action": "comment_added",  "by": "u_yyy", "by_name": "Op. Verdi",    "at": "..." },
  { "action": "status_changed", "by": "u_yyy", "by_name": "Op. Verdi",    "at": "...", "note": "in lavorazione → risolto" }
]

Comments & File Uploads

Comments are stored in data/comments.json separately from tickets, linked by ticket_id.

  • Any participant (user or staff) can comment on open tickets
  • Users cannot comment on chiuso tickets
  • Each comment can carry multiple image attachments uploaded in the same request
  • Uploaded files are saved in uploads/<ticket_id>/<filename>
  • Accepted types: image/jpeg, image/png, image/gif, image/webp — max 5 MB per file

Email Notifications

Triggered automatically by API actions when Notifiche email is enabled in settings.

EventWho receives it
New ticket createdAll Admins + Operators
Ticket status changedAll Admins + Operators + Ticket opener
Comment added by a UserAll Admins + Operators (not the commenter)
Comment added by Operator/AdminTicket opener (not the commenter)

Emails are HTML-formatted with the brand color, a summary table, and a direct link to the ticket. If SMTP credentials are configured, the built-in SMTP client handles delivery (TLS/SSL/plain); otherwise falls back to PHP's mail().

Notifications are sent inline with the API response (not queued). On very busy systems, consider moving them to a background process or cron job to avoid response latency.

API Reference

All API calls go to inc/api.php?action=<name>. GET requests pass parameters as query strings; POST/mutating requests send a JSON body.

Authentication

POSTapi.php?action=login
POSTapi.php?action=logout
GETapi.php?action=me

Tickets

GETapi.php?action=tickets[&status=&priority=&category=&search=&assigned_to=&page=&per_page=]
GETapi.php?action=ticket&id=<id>
POSTapi.php?action=create_ticket
POSTapi.php?action=update_ticket operator+admin
POSTapi.php?action=delete_ticket admin only

Comments & Uploads

GETapi.php?action=comments&ticket_id=<id>
POSTapi.php?action=add_comment multipart/form-data with optional uploads[]
POSTapi.php?action=upload multipart/form-data

Users admin only

GETapi.php?action=users
POSTapi.php?action=create_user
POSTapi.php?action=update_user
POSTapi.php?action=delete_user
GETapi.php?action=operators operator+admin

GETapi.php?action=report
GETapi.php?action=logs[&page=]
GETapi.php?action=notifications

SMTP Setup

Configure from ⚙️ Impostazioni → Notifiche Email. Common provider settings:

Gmail (App Password required)

SMTP Host:       smtp.gmail.com
Port:            587
Encryption:      TLS
Username:        youraddress@gmail.com
Password:        <16-char App Password from Google Account → Security → 2FA → App passwords>
From Email:      youraddress@gmail.com

Outlook / Microsoft 365

SMTP Host:       smtp.office365.com
Port:            587
Encryption:      TLS
Username:        youraddress@yourdomain.com
Password:        <account password>

Generic Provider (e.g. Aruba, Hosting.it)

SMTP Host:       smtps.aruba.it   (or as specified by your host)
Port:            465
Encryption:      SSL
Username:        youraddress@yourdomain.it
Password:        <mailbox password>
If SMTP is not configured, the system falls back to PHP's mail() function. This may work on some shared hosting but is often blocked or sent to spam. SMTP is strongly recommended for reliable delivery.
